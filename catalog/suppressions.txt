# --- glibc/TSan false positives (uninstrumented libc-internal locks) ---
# _multiprocessing.SemLock create/destroy: SemLock() -> sem_open() -> tsearch() (insert into
# glibc's process-global __sem_mappings tree) races SemLock dealloc -> sem_close() -> tdelete()
# (remove). glibc serializes every tree access with its internal __sem_mappings_lock (an lll/futex
# lock TSan does not interpose), so it's a false positive, not a CPython use-after-free (the two
# SemLock PyObjects are distinct + correctly refcounted). Same class as the tzset false positive.
# Confirmed by glibc-2.43 disassembly; see reports/TSAN-0003-multiprocessing-semlock-dealloc/.
Modules/_multiprocessing/semaphore.c:_multiprocessing_SemLock_impl \| Objects/object.c:_Py_Dealloc

# Concurrent __init__ of a SHARED object -- unsupported per cpython#127192.
#
# WITHDRAWN 2026-07-16: this suppression was based on a PARSER ARTIFACT. fusil's tsan_dedup
# ACCESS regex did not match the "Previous atomic write" stanza header, so the 2nd stanza was
# dropped and the 1st duplicated -- fabricating the symmetric
# `bytearray___init___impl | bytearray___init___impl` signature that looked like two threads
# re-initialising one shared bytearray. With the parser fixed, those same 10 vehicles resolve to
#   Include/cpython/pyatomic_gcc.h:_Py_atomic_store_ptr_release | Objects/bytearrayobject.c:bytearray___init___impl
# i.e. `bytearray(shared_list)` READING the source list while another thread appends to it
# (list_resize's atomic ob_item publish) -- a shared-LIST reader face of TSAN-0013, which is a
# real bug (Yhg1s ruling), NOT concurrent __init__. Suppressing it was hiding a genuine finding.
# Nothing from the concurrent-__init__ class is suppressed anymore; re-add only if a *true*
# (post-fix) signature genuinely shows two __init__ sites racing.

# --- fleet-03 additions (2026-07-16) ---

# glibc/libc-internal false positive: time.tzset()/time.mktime() -> glibc tzset_internal free()s and
# strdup()s tzname[] under glibc's own uninstrumented tzset_lock (an lll lock TSan can't see). The
# racing CPython frame collapses to the generic cfunction_vectorcall_NOARGS on BOTH sides because
# parse_report strips the libc (free/tzset_internal) frames. Same class as TSAN-0003 / the tzset
# reproducer; confirmed a pure-C glibc repro (no Python) also races. Anchored to the both-collapsed
# form so a real NOARGS-cfunction race (which would carry a distinct frame on one side) still surfaces.
# Proper long-term fix is teaching tsan_dedup.parse_report to keep the innermost libc frame.
^Objects/methodobject.c:cfunction_vectorcall_NOARGS \| Objects/methodobject.c:cfunction_vectorcall_NOARGS$

# Foreign-C-library false positive: OpenSSL. Concurrently building ssl.SSLContext objects races
# inside libcrypto (memcmp <- libcrypto.so.3 <- tp_new_wrapper); each thread makes its OWN SSLContext,
# the shared state is an OpenSSL-internal cache, not CPython's. See TSAN-0020. Nearest symbolized
# CPython frame collapses to tp_new_wrapper on both sides (libcrypto is stripped). Anchored to the
# both-collapsed form. Not CPython's to fix.
^Objects/typeobject.c:tp_new_wrapper \| Objects/typeobject.c:tp_new_wrapper$

# --- fleet-11 additions (2026-07-18) ---

# tracemalloc: fusil fuzzes the tracemalloc module concurrently, but tracemalloc.start()/stop()
# swaps the PROCESS-GLOBAL allocator via PyMem_SetAllocator, which inherently races every other
# thread's PyMem_GetAllocator / allocation. tracemalloc is a debug tool, not a free-threading
# target, and you must not start/stop it concurrently with other threads allocating -- expected
# noise, not a target finding. Matches any race with a tracemalloc.c frame or the allocator swap.
tracemalloc\.c:
Objects/obmalloc\.c:PyMem_(Set|Get)Allocator
